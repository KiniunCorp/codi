# CODI Operations Runbook

This runbook captures day-2 operational procedures for CODI across local CLI usage, the Slim container, and the Complete container with embedded llama.cpp runtime and LoRA adapter support.

## 1. Artefact Locations

- Runs and reports: `runs/` (override via `CODI_OUTPUT_ROOT` or CLI `--out`).
- Model weights: `/models/weights/` (`MODEL_MOUNT_PATH`).
- LoRA adapters: `/models/adapters/<adapter-id>/` (`ADAPTER_PATH`).
- Environment snapshot: `runs/<id>/metadata/environment.json`.
- CPU sanity report: `runs/<out>/perf/cpu_perf_report.json` (generated by `codi perf`).

## 2. Health Checks

| Component | Probe | Expected |
|-----------|-------|----------|
| FastAPI service | `GET http://<host>:8000/healthz` | `{  "status": "ok" }` |
| Local LLM server | `GET http://<host>:8081/healthz` | `{  "status": "ok", "model_id": "..." }` |
| LLM rank endpoint | `POST http://<host>:8000/llm/rank` (Complete) | `{"adapter_version": "...", "ranking": [...]}` |
| Container | `docker inspect --format '{{ .State.Health.Status }}' <container>` | `healthy` |
| Adapter mount | `python3 /opt/codi/scripts/verify_adapter.py $ADAPTER_PATH` | Exit code 0 |

If health checks fail:
1. Collect logs: `docker logs <container>`
2. Check runtime logs: `/work/runs/_runtime.log` inside Complete container
3. Verify adapter: `docker exec <container> ls -la /models/adapters/`
4. Check startup logs for adapter version and status

## 3. Start & Stop Procedures

### Local CLI

```bash
python3 -m cli.main run demo/node --out runs/local
python3 -m cli.main report runs/local/<run-id>
```

- Keep `AIRGAP=true` unless testing vetted endpoints via `AIRGAP_ALLOWLIST`.
- Run `python3 -m cli.main perf --out runs/perf` to capture CPU sanity metrics.

### Slim Container

```bash
docker run --rm -it -v "$PWD:/work" -p 8000:8000 codi:slim
```

### Complete Container (with Adapter)

```bash
docker run --rm -it \
  -v "$PWD:/work" \
  -v "$HOME/.codi-models/adapters:/models/adapters" \
  -e AIRGAP=true \
  -e LLM_ENABLED=true \
  -e CODE_MODEL=qwen2.5-coder-1.5b \
  -e ADAPTER_PATH=/models/adapters/qwen15b-lora-v0.1 \
  -p 8000:8000 -p 8081:8081 \
  codi:complete
```

The runtime boot script:
1. Validates adapter mount and reads metadata
2. Logs adapter version and configuration
3. Starts llama.cpp with adapter (if present)
4. Launches FastAPI with LLM endpoints enabled

Post-boot validation:
1. Check the adapter matrix via `docker exec <container> cat /work/patterns/rules.yml | grep codi.ruleset_version`.
2. Hit the rank endpoint without leaving the container network:
   ```bash
   docker exec <container> python3 -m cli.main llm rank demo/node
   ```
   Ensure the response includes `adapter_version`, `ruleset_version`, and deterministic ranking payloads.
3. Record the promotion reference from [`docs/llm_promotion_checklist.md`](llm_promotion_checklist.md) when rolling new adapters.

## 4. Configuration Toggles

| Variable | Description | Default |
|----------|-------------|---------|
| `AIRGAP` | Blocks outbound HTTP/S via `core.security.enforce_airgap_guard`. | `true` |
| `AIRGAP_ALLOWLIST` | Comma-separated host allowlist (supports wildcards). | _empty_ |
| `LLM_ENABLED` | Enables/disables assist summaries. | `false` (Slim) / `true` (Complete) |
| `LLM_ENDPOINT` | Override embedded LLM endpoint. | `http://127.0.0.1:8081` |
| `CODE_MODEL` | Base code model identifier. | `qwen2.5-coder-1.5b` |
| `ADAPTER_PATH` | Path to LoRA adapter directory. | `/models/adapters/qwen15b-lora-v0.1` |
| `ADAPTER_VERSION` | Adapter version (auto-detected from metadata). | `unknown` |
| `LLAMA_CPP_THREADS` | CPU threads for llama.cpp inference. | `4` |
| `MODEL_MOUNT_PATH` | Location of mounted weights and adapters. | `/models` |
| `CODI_RULESET_VERSION` | Propagated ruleset tag emitted by rendered templates (`LABEL codi.ruleset_version`). | `2025.11-llm` |
| `RULES_PATH` | Override template file. | `patterns/rules.yml` |
| `CODI_OUTPUT_ROOT` | Run artefact root. | `runs/` |

## 5. Standard Operating Procedures

### Rotate Runs

```bash
# Keep the 30 most recent runs
ls -dt runs/* | tail -n +31 | xargs -r rm -rf
```

### Verify Security Gates

- Execute: `python3 -m pytest tests/test_security.py` (see `docs/security_verification.md`).
- Confirm blocked builds present clear `SecurityPolicyError` messages in CLI/API responses.

### Refresh Adapters

1. Train or obtain new adapter weights (see `training/qwen15b_lora/` for the QLoRA pipeline).
2. Copy adapter to mounted directory:
   ```bash
   cp -r training/qwen15b_lora/output/* ~/.codi-models/adapters/qwen15b-lora-v0.1/
   ```
3. Verify adapter integrity:
   ```bash
   python3 docker/scripts/verify_adapter.py ~/.codi-models/adapters/qwen15b-lora-v0.1
   ```
4. Restart the Complete container (`docker restart <container>` or rerun the command above).
5. Check startup logs for adapter version confirmation.
6. Verify in `runs/<id>/metadata/environment.json`.

### Refresh Base Models

1. Copy new GGUF weights into the mounted `/models/weights/` directory.
2. Update `CODE_MODEL` environment variable if using a different model.
3. Restart the Complete container.
4. Check `runs/<id>/metadata/environment.json` to confirm configuration.

### Performance Sanity

- Run: `python3 -m cli.main perf --out runs/perf --analysis-budget 5 --total-budget 180`.
- Review `runs/perf/cpu_perf_report.json` or `docs/performance_cpu_sanity.md` for latest metrics.

## 6. Troubleshooting

| Symptom | Resolution |
|---------|------------|
| `AirgapViolation` raised for internal host | Add host to `AIRGAP_ALLOWLIST` or disable air-gap temporarily (for controlled testing). |
| LLM assist disabled unexpectedly | Check `LLM_ENABLED` in `environment.json`; ensure adapter mounted when running Complete image. |
| Adapter not loading | Verify: (1) `ADAPTER_PATH` points to valid directory, (2) required files present, (3) checksums valid. Run `verify_adapter.py`. |
| Checksum mismatch | Re-copy adapter files or update `metadata.json` checksums. Remove `checksums` section for development. |
| `llama-server: command not found` | Rebuild Complete container: `make build-complete`. Verify binary: `docker run codi:complete which llama-server`. |
| Missing artefacts | Verify `CODI_OUTPUT_ROOT` is writable and not on a read-only volume. |
| CLI cannot locate Dockerfile | Ensure target directory contains `Dockerfile` or `docker/Dockerfile`; adjust path accordingly. |
| Adapter version shows "unknown" | Add `metadata.json` to adapter directory with `version` field. See `/models/adapters/qwen15b-lora-v0.1/` for example. |

## 9. Release Publishing & Rollback

This section covers the reproducible workflow for publishing signed container images to GHCR (`ghcr.io/<namespace>/codi-slim` and `ghcr.io/<namespace>/codi-complete`).

### Pre-flight Checklist
1. `make test` (or `pytest`) passes on the release commit.
2. `make release-images RELEASE_VERSION=vX.Y.Z IMAGE_NAMESPACE=<org>/<repo>` succeeds locally (loads images into Docker).
3. You are logged in to the target registry (`docker login ghcr.io`).
4. `REGISTRY`, `IMAGE_NAMESPACE`, and `RELEASE_VERSION` are agreed upon with the release owner.

### Publish Flow
```bash
# Optional: dry-run build (loads images locally without pushing)
make release-images RELEASE_VERSION=v1.4.0 IMAGE_NAMESPACE=my-org/codi

# Publish directly from a workstation (push=true). Requires docker login + write perms.
make publish-images \
  RELEASE_VERSION=v1.4.0 \
  IMAGE_NAMESPACE=my-org/codi \
  REGISTRY=ghcr.io

# Create an annotated git tag and push it to trigger CI
git tag -a v1.4.0 -m "CODI v1.4.0 release"
git push origin v1.4.0
```

The `.github/workflows/release-images.yml` pipeline will:
- Derive the registry namespace (default `<owner>/<repo>`)
- Build Slim + Complete images with BuildKit
- Push `vX.Y.Z`, `latest`, and digest tags for each image
- Generate SPDX SBOMs (`sbom/slim-<version>.spdx.json`, `sbom/complete-<version>.spdx.json`)
- Sign images and attach SBOM attestations using cosign keyless (OIDC)
- Upload SBOMs as workflow artifacts (`sbom-<version>.zip`)

Monitor the workflow run to confirm each job finishes successfully. Upon completion, copy the Slim/Complete digests into release notes.

### Verification Commands
```bash
OWNER=my-org
REPO=codi
VERSION=v1.4.0
IMAGE=ghcr.io/$OWNER/$REPO/codi-slim:$VERSION

# Signature verification (keyless / OIDC)
cosign verify \
  --certificate-identity "https://github.com/${OWNER}/${REPO}/.github/workflows/release-images.yml@refs/tags/${VERSION}" \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  "$IMAGE"

# SBOM attestation verification (pulls from registry automatically)
cosign verify-attestation \
  --type spdxjson \
  "$IMAGE"
```

Download the `sbom-<version>` artifact from the workflow run if you need offline copies of the SPDX files.

### Rollback & Hotfix Strategy
1. Identify the last known-good tag (e.g., `v1.3.2`) and confirm its digests via `cosign verify`.
2. Update staging/production deployments to point to the previous tag:
   ```bash
   export CODI_TAG=v1.3.2
   docker pull ghcr.io/$OWNER/$REPO/codi-slim:$CODI_TAG
   docker pull ghcr.io/$OWNER/$REPO/codi-complete:$CODI_TAG
   ```
3. If `latest` needs to be re-pointed, tag locally and re-publish:
   ```bash
   docker tag ghcr.io/$OWNER/$REPO/codi-slim:$CODI_TAG ghcr.io/$OWNER/$REPO/codi-slim:latest
   docker tag ghcr.io/$OWNER/$REPO/codi-complete:$CODI_TAG ghcr.io/$OWNER/$REPO/codi-complete:latest
   make publish-images RELEASE_VERSION=$CODI_TAG IMAGE_NAMESPACE=$OWNER/$REPO
   ```
4. For hotfixes, cherry-pick the fix, bump `RELEASE_VERSION` (e.g., `v1.4.1`), rerun the publish flow above, and document the new digests in the rollout log.

Dependencies:
- Docker Buildx (local + CI)
- `sigstore/cosign` (installed automatically in CI; install locally via `brew install cosign` or `go install sigstore.dev/cosign/cmd/cosign@latest`)
- Access to the GHCR namespace with `packages:write`
